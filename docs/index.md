# GeneSys
# Интерактивный конструктор живого
Цель проекта - создать консольное приложение, позволяющее пользователю работать с базой данных сайта **kegg.jp** в интерактивном виде.\
Это необходимо для исследования известных путей взаимодейтсвия (*pathways*) и создания своих проектов, для потенциальной мокрой биологии (*химические реакции в рамках пробирок/компартментов/клеток/популяций*)

# Архитектура проекта
Проект подразделяется на несколько глобальных модулей:
- CLI (интерфейс командной строки)
- Бизнес-логика (алгоритмы работы с веществами и т.д.)
- База данных (создание/хранение/извелечение)
- Парсинг (сбор данных с различных* ресурсов)

## CLI
Программа запускается из файла **.exe** и работает в консоли в рамках одного **loop**. Имеется множество состояний, в зависимости от которых ожидается ввод/совершается вывод данных в консоль.\
Например, при только что запущенном приложении, программа выводит список проектов и ожидает ввода названия/номер проекта. Все это отслеживается через машину состояний, в рамках которой и происходит перемещение пользователя по программе.
### Используемые подходы
В нашем случае будет использоваться подход "Состояние-Структура", где каждое состояние реализует общий трейт (интерфейс).

## Бизнес-логика
Представляет из себя алгоритмы по работе с веществами, их наборами (компартментами) и клетками (наборами компартментов).\
Подразумеваются следующие блоки **функционалов объектов**:
- Вещества:
  - Информация о веществе
  - ...
- Компартменты:
  - Создание
  - Наполнение
  - Построение древа реакций
  - Рассчет "субстрата" или "продукта"
  - Построение цепочек взаимодействия
  - Внедрение/Изъятие веществ
  - Динамическое изменение состава
  - ...
- Межкомпартментное взаимодействие:
  - Пути перемещения веществ между компартментами
  - ...

Также необходим следующий **общий функционал**:
- Компиляция проекта:
  - Построение таймлайна изменений
  - Создание "слоев" для отслеживания изменения в их рамках.
  - Поиск циклических взаимодейтсвий

## База данных

Хранение всей информации предполагается локально у пользователя посредством встроенной базы данных **SQLite**. Она должна имеет следующие данные:
- Пользователи
- Проекты
- Компартменты
- Статические данные:
  - Энзимы
  - Простые вещества
  - Реакции
  - ...

В рамках каждого блока данных возможно варьирование на один-несколько файлов.\
Полная обвязка необходимыми методами **поиска/извлечения/модификации/добавления** данных

## Парсинг
Пополнение данными происходит посредством парсинга через **API** или **Plane Text** соответствующих сайтов. Необходимые требования:
- Стандартизация
- Валидация
- Контроль ограничений API взаимодействия
